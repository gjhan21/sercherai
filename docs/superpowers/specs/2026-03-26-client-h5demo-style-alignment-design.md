# Client H5Demo Style Alignment Design

**日期：** 2026-03-26  
**范围：** `/Users/gjhan21/cursor/sercherai/client`

## 目标

将 `client/h5demo` 中已经确认的 PC / H5 页面设计语言，对接到当前真实 `client` 运行时页面中。

本轮目标不是“把静态 demo 原样搬进项目”，而是：

- 让真实页面长成 `h5demo` 的样子
- 保留现有真实 API、登录态、会员、实名、消息、评论、举报、关注与资讯链路
- 把没有对应设计稿的 `community` 三页，一并收敛到同一套中文金融站风格和产品口吻

## 背景

当前 `client` 已经有一套真实运行时页面和全局样式骨架：

- 入口：`client/src/main.js`
- 页面：`client/src/views`
- 全局样式：`client/src/styles/tokens.css`、`base.css`、`finance-shell.css`、`finance-pages.css`
- 真实主壳：`client/src/components/ClientLayout.vue`

同时，仓库里也已经存在一套新的静态设计稿：

- `client/h5demo/home-*.html`
- `client/h5demo/strategies-*.html`
- `client/h5demo/watchlist-*.html`
- `client/h5demo/news-*.html`
- `client/h5demo/archive-*.html`
- `client/h5demo/membership-*.html`
- `client/h5demo/profile-*.html`
- `client/h5demo/auth-*.html`

这些设计稿已经明确了更符合中文用户习惯的金融站阅读链、模块优先级和蓝金金融系视觉语言，但它们仍然只是静态 HTML。

当前真实页面虽然已经有部分金融站样式基础，但仍存在以下问题：

- 页面之间视觉和结构完成度不一致
- 顶部、主体、底部的风格统一性不够稳定
- 一些页面仍带有“研发说明感”或“工具页感”
- `community` 虽已具备真实能力，但与主站风格和语气仍有割裂

## 用户已确认的范围

- 采用“共享壳层 + 页面重排”方案，而不是只换肤或整页近乎重写
- `client/h5demo` 只作为设计稿来源，不直接作为运行时代码使用
- 当前正式入口继续保持 `client/src/main.js`
- 当前真实全局样式体系继续保留，并作为本轮统一收敛入口
- `client/src/apps/h5`、`client/src/apps/pc` 只作为局部参考，不替换当前主入口
- 有设计稿的 8 个页面全部要对齐
- 没有设计稿的 `community` 三页也要按新设计稿风格一起调整
- `community` 的调整深度为：
  - 视觉统一
  - 信息架构重排
  - 交互文案与产品口吻收口

## 设计原则

### 1. 真实页面优先，不复制静态 HTML

- 不将 `client/h5demo/*.html` 直接引入 Vue 运行时
- 不把 demo 里的静态结构硬塞进当前页面
- 只提炼 demo 中的视觉语言、模块顺序、版式节奏和文案口吻
- 在真实页面模板、真实数据和真实交互基础上完成重构

### 2. 真实业务能力边界不变

- 不改后端 API 契约
- 不改真实路由含义
- 不改认证、会员、实名、消息、评论、举报、发帖等既有业务规则
- 不为了贴合 demo 新增不存在的功能模块

### 3. 优先统一阅读链，而不是只统一颜色

页面改造的核心不是“同色系”，而是让用户在进入页面第一屏时立刻明白：

- 这页最重要的任务是什么
- 先看什么
- 再看什么
- 最后该去哪里继续处理

### 4. PC 与窄屏共用同一套真实页面

- 本轮不新开第二套真实 PC / H5 路由工程
- 当前真实页面继续承担桌面和窄屏展示
- 桌面端体现版心、双栏、侧栏和信息密度
- 窄屏下自然压缩为单列阅读顺序
- 目标是“同一套页面兼顾 PC 与 H5 阅读感”，而不是“再做两套运行时”

### 5. `community` 是主站内容链的延伸，不是独立聊天室产品

- `community` 只承接围绕股票、期货、资讯、策略对象的结构化观点讨论
- 不做即时聊天刷屏
- 不做独立社交产品语气
- 要与首页、策略、资讯、档案形成站内闭环

## 非目标

- 不切换 `client` 的正式入口到 `apps/h5` 或 `apps/pc`
- 不额外建立新的共享业务层、状态管理层或跨端工程层
- 不对 `client/h5demo` 做运行时集成
- 不引入新的路由或新的后端能力
- 不把社区扩展成图片、附件、聊天室、直播或开放广场系统

## 总体方案

本轮采用“共享壳层 + 页面重排”方案。

整体执行结构固定为：

1. 先统一共享壳层和全局金融站视觉语言
2. 再按 `h5demo` 重排 8 个主页面的信息结构
3. 最后把 `community` 三页一起收进同一套阅读链和产品口吻

这样做的原因是：

- 风险比整页近乎重写更可控
- 更适合当前主分支直接连续推进
- 能保住真实数据流、真实写操作和真实权限链
- 最终效果仍能明显向 `h5demo` 靠拢，而不是停留在“局部换皮”

## 文案与语气规范

本轮统一清理以下类型文案：

- “当前版本只支持……”
- “生产页不保留……”
- “开发中 / demo / 试验中” 这类对生产阅读无帮助的话术

正式页面主体统一改成产品口吻：

- 页面定位
- 当前任务
- 权限承接
- 风险提示
- 下一步动作

可以保留的说明，前提是它们具有产品意义：

- 风险提示
- 权限边界
- 阅读顺序提示
- 实名 / 会员 / 登录状态影响

## 验收标准

### 视觉验收

- 顶部、主体、底部属于同一套蓝金金融站视觉系统
- 按钮、标签、卡片、表格、说明条风格统一
- 不再出现页面间明显的“换皮感”或“多套系统拼接感”

### 结构验收

- 每个页面首屏都明确表达主任务
- 阅读顺序符合中文金融站 / 金融 App 习惯
- 下一步入口清晰，不让用户停在无动作页面

### 行为回归

- 登录 / 注册 / redirect / invite_code 不回退
- 资讯搜索 / 正文 / 附件 / 权限承接不回退
- 会员下单 / 状态 / 实名激活不回退
- 我的页消息 / 订阅 / 邀请 / 实名不回退
- 关注 / 变化 / 版本对照不回退
- 社区发帖 / 评论 / 点赞 / 举报不回退

### 工程验收

- 批次改造后执行 `cd client && npm run build`
- 只在 3 个以上页面重复时才抽共享样式块
- 不引入新的路由体系和业务协议层

## 风险与控制

### 风险 1：为了贴 demo 过度重写页面

控制方式：

- 不直接复制 demo HTML
- 优先保留真实模板中的数据流和事件处理
- 只重构承载层和模块顺序

### 风险 2：共享层抽象过早

控制方式：

- 先改页面，再观察重复度
- 只有真正高频复用的壳层和基础块进入共享样式

### 风险 3：community 再次脱离主站

控制方式：

- community 与 8 个主页面共用同一套壳层语言
- 文案改成产品口吻
- 入口和回流关系始终围绕首页 / 策略 / 资讯 / 档案

## 结论

本轮正式方向已经固定：

- 以 `client/h5demo` 为设计稿基线
- 以 `client/src` 真实页面为运行时基线
- 采用“共享壳层 + 页面重排”方式完成风格与结构对接
- 将 `community` 三页一并纳入同一套中文金融站视觉、阅读链和产品语气

这会让当前 `client` 不再只是“已有功能页 + 局部金融化样式”，而是收敛成一套更完整、更统一、也更符合中文用户阅读习惯的真实客户端。
