# New Client Fullsite Demo Design

## 背景

`newclient` 是新客户端目录，当前阶段先做设计 demo，不进入真实前端开发。

`newclient/h5demo` 作为本轮静态 HTML demo 输出目录，必须严格建立在当前主项目 `client` 已有的真实 API、真实功能和真实路由能力之上。首页 demo 已经确认方向，本 spec 用于把“首页之后的整套页面体系”一次性收口，避免后续页面各做各的、风格和信息架构失控。

本 spec 依赖以下真相源文档：

- `newclient/h5demo/现有client接口与功能盘点.md`
- `docs/superpowers/specs/2026-03-24-newclient-homepage-demo-design.md`

## 设计目标

本轮全站 demo 的目标不是做一套“理想化新产品”，而是做一套可落回当前真实能力的前端页面方案。

目标固定为：

- PC 端更像中文金融站 / 券商研究终端
- H5 端更像国内金融 App 首页与内容客户端
- PC 与 H5 共享同一套蓝金金融视觉系统
- PC 与 H5 不做完全对称复制，而采用“同内容骨架、不同端表达方式”
- 首页、策略、关注、资讯、档案、会员、我的、登录注册八类页面形成完整闭环

## 设计范围

本轮纳入设计的页面如下：

1. `home`
2. `strategies`
3. `watchlist`
4. `news`
5. `archive`
6. `membership`
7. `profile`
8. `auth`

说明：

- `auth` 不属于主业务导航页，但属于必须设计的身份入口页
- 与旧版 H5 不同，本轮新客户端中 `watchlist` 与 `archive` 在 H5 端也规划为独立页面

## 统一设计原则

### 1. 视觉原则

- 全站统一为蓝金金融系，不回到暖绿 / 米色主题
- PC 强调“中文金融站 / 研究终端”感，不做欧美 SaaS Landing 风
- H5 强调“国内金融 App 首页 / 内容 App”感，不做简化网页风
- 标题、说明、标签、底部信息都要属于同一套色系与语言系统

### 2. 信息原则

- 所有页面都强调“先看结论，再看理由，再继续阅读”
- 推荐和研报都必须是明确层级，不互相淹没
- 资讯、推荐、关注、档案之间必须形成阅读链，不是零散页面拼接
- 首页不等于全功能，重点是导入主阅读链

### 3. 边界原则

本轮不新增以下内容能力：

- 大盘指数横排实时条
- 热门题材榜
- 社区 / 发帖 / 讨论区 / 聊天室
- 直播、短视频
- 开放平台
- 云端自选股或社交组合
- 任何当前 `client` 未体现的后端新能力

允许在 demo 中组织表达的真实能力包括：

- 股票推荐
- 期货策略 / guidance 承接
- 市场事件
- 资讯、研报、期刊、正文、附件、权限锁
- 历史推荐样本与复盘
- 本地关注与变化追踪
- 会员状态、配额、订单、支付、实名激活
- 账户、消息、邀请、实名、订阅等用户中心能力

## 全站导航关系

## PC 主导航

PC 主导航固定为：

- 首页
- 策略
- 关注
- 资讯
- 档案
- 会员
- 我的

说明：

- PC 端允许 `archive` 与 `watchlist` 都作为顶级可见入口存在
- `auth` 不在主导航中，由登录态和权限链承接

## H5 底部导航

H5 底部 Tab 固定为：

- 首页
- 策略
- 关注
- 资讯
- 我的

说明：

- `archive` 不放入底部 Tab
- `membership` 不放入底部 Tab
- `auth` 不放入底部 Tab

## H5 二级进入路径

### `archive`

从以下入口进入：

- 首页快捷入口
- 首页历史样本预览卡
- 资讯页头部二级入口
- 关注页中的“去完整档案”
- 策略页中的“看历史样本 / 看档案”

### `membership`

从以下入口进入：

- 首页快捷入口
- 资讯页正文 / 附件 / 研报权限 CTA
- 策略页深读与更多内容 CTA
- 我的页中的会员状态卡
- 未登录或未激活的权限承接区

## 整体用户阅读链

全站页面关系按以下主链路组织：

`首页 -> 策略 / 资讯 -> 关注 -> 档案 -> 会员 / 我的 -> 登录注册`

角色理解如下：

- `home`：今天先看什么
- `strategies`：为什么推荐、怎么执行
- `news`：从导读进入深读
- `watchlist`：加入后怎么持续跟踪变化
- `archive`：历史样本怎么复盘
- `membership`：为什么升级、如何生效
- `profile`：账户状态和待办怎么处理
- `auth`：高效率身份入口

## 页面级设计规范

## 1. 首页 `home`

### 页面主任务

让用户一进来就知道“今天先看什么”，并在首屏同时建立“推荐主位”和“焦点研报”两条主线。

### PC 结构

- 顶部站点导航
- 推荐主 Hero
- 为什么选择该推荐
- 历史推荐样本预览
- 焦点研报卡
- 市场资讯导流
- 底部信息区

### H5 结构

- 顶部品牌头与搜索 / 消息入口
- 主线 Hero
- 快捷入口宫格
- 轻量数据条
- 推荐主卡
- 推荐理由 / 风控说明
- 焦点研报大卡
- 历史样本预览
- 市场资讯列表
- 精简底部说明
- 底部 Tab

### 不允许做成的样子

- 欧美 SaaS Landing 风
- 纯营销页
- 普通资讯门户首页
- 缺少底部信息的半截页面

## 2. 策略页 `strategies`

### 页面主任务

把“推荐是什么、为什么成立、怎么执行、风险边界在哪里”讲清楚。

### PC 结构

- 页面定位区
- 主推荐工作台
- 为什么推荐
- 执行区间与风险边界
- 股票 / 期货 / 市场事件分层内容区
- 右侧阅读顺序与页面关系说明

### H5 结构

- 页面标题与筛选切换
- 推荐主卡
- 推荐理由卡
- 风险边界卡
- 观点流列表
- 期货机会与事件卡
- 底部 Tab

### 不允许做成的样子

- 只有股票列表
- 后台筛选台
- 被资讯流稀释的伪推荐页

## 3. 关注页 `watchlist`

### 页面主任务

回答“我关注的对象最近发生了什么变化，下一步该做什么”。

### PC 结构

- 页面定位区
- 高优先关注样本
- 关注主列表
- 变化拆层说明
- 右侧回访节奏、能力范围、页面关系

### H5 结构

- 页面头与最近更新时间
- 今日优先回访卡
- 变化速览卡组
- 关注列表流
- 去策略 / 资讯 / 档案的下一步入口
- 底部 Tab

### 不允许做成的样子

- 纯自选行情页
- 只有关注对象名称、没有变化信息
- 云端社交自选股页

## 4. 资讯页 `news`

### 页面主任务

从导读进入深读，尤其要把研报从普通新闻里独立出来。

### PC 结构

- 页面定位与焦点研报
- 分类切换区
- 主列表区
- 正文详情区
- 附件与权限承接区
- 右侧栏目、权限、阅读链说明

### H5 结构

- 顶部标题、搜索、分类切换
- 焦点研报卡
- 主文章卡
- 内容流列表
- 正文详情模式
- 附件与权限 CTA
- 底部 Tab

### 不允许做成的样子

- 内容瀑布流
- 研报和新闻完全同层级
- 只剩列表、没有深读结构

## 5. 档案页 `archive`

### 页面主任务

回答“过去推荐了什么、结果如何、为什么止盈止损、后来怎么变化”。

### PC 结构

- 档案定位区
- 筛选与概览
- 历史推荐时间线
- 样本复盘卡
- 版本差异与变化说明
- 右侧复盘方法与阅读顺序

### H5 结构

- 页面头与筛选条
- 焦点历史样本卡
- 历史样本流
- 单条样本详情
- 历史理由、涨跌结果、后期处理原因、版本变化

### 不允许做成的样子

- 只有结果没有理由
- 只有成功样本
- 只做涨跌排行页

## 6. 会员页 `membership`

### 页面主任务

解释“为什么升级、当前状态是什么、下一步怎么完成”。

### PC 结构

- 当前状态卡
- 今日来到会员页的承接原因
- 能力升级解释
- 套餐与能力对比
- 配额与订单状态
- 右侧待办、会员摘要、实名状态

### H5 结构

- 收银台头部与状态条
- 主推套餐卡
- 升级价值解释卡
- 套餐选择区
- 支付通道与激活区
- 订单状态区

### 不允许做成的样子

- 纯价格页
- 营销长页
- 把实名激活压到页面底部

## 7. 我的页 `profile`

### 页面主任务

让用户先看到今天要处理什么，再进入账户确认与管理。

### PC 结构

- 账户身份概览
- 今日行动板
- 待办中心
- 查询中心
- 快捷入口
- 右侧账户摘要与待处理事项

### H5 结构

- 账户主卡
- 今日待办
- 常用入口宫格
- 账户服务卡流
- 消息 / 邀请 / 实名区块
- 底部 Tab

### 不允许做成的样子

- 纯资料编辑页
- 后台查询语气
- 隐藏会员与实名状态

## 8. 登录注册页 `auth`

### 页面主任务

让用户快速完成身份进入，并且保留来源回跳。

### PC 结构

- 顶部来源说明
- 登录 / 注册双栏主区
- 当前支持能力说明
- 登录后回跳说明

### H5 结构

- 顶部返回与品牌区
- 登录 / 注册单列主区
- 邀请码状态区
- 来源回跳说明
- 当前支持能力说明

### 不允许做成的样子

- 社交登录广场
- 营销长页
- 登录后丢失来源回跳

## 推荐推进批次

### 第一批：主阅读链

- `strategies`
- `watchlist`
- `news`

### 第二批：复盘与转化链

- `archive`
- `membership`

### 第三批：账户与身份链

- `profile`
- `auth`

## 推荐执行顺序

建议严格按以下顺序出图：

1. `strategies`
2. `watchlist`
3. `news`
4. `archive`
5. `membership`
6. `profile`
7. `auth`

原因：

- 最贴近首页之后的真实用户使用链
- 最容易保持风格统一
- 可以先把“内容主链”做稳，再做“复盘和账户支持链”

## 联动审查建议

以下页面需要联动审查，而不是独立看：

### 第一组

- `home + strategies + watchlist`

重点看：

- 首页导到策略是否自然
- 策略导到关注是否顺畅
- 关注页是否真承接“变化”

### 第二组

- `home + news + archive`

重点看：

- 首页焦点研报和资讯页是否是同一语言
- 历史样本预览与档案页是否是同一套复盘逻辑

### 第三组

- `news + membership + auth`

重点看：

- 权限锁到会员页是否自然
- 会员页到登录页是否顺
- 登录后回跳说明是否清楚

### 第四组

- `membership + profile`

重点看：

- 会员状态、实名状态、订单状态是否统一
- 两页是否属于同一套账户系统

## 审查重点

审查整套全站 demo 时，主要看以下事项：

1. 首页风格是否真正延续到其他页面
2. PC 是否始终保持中文金融站 / 研究终端气质
3. H5 是否始终保持国内金融 App 气质
4. 推荐、研报、关注、档案、会员、我的之间是否形成清晰阅读链
5. 是否严格遵守当前真实能力边界
6. 是否仍然保持蓝金金融系的一致性

## 下一步建议

本 spec 确认后，后续不建议七页齐出，而应从第一批开始推进。

最合理的下一个设计动作是：

- 先做 `strategies` 的 PC / H5 demo
- 以首页为基准建立第二个页面锚点
- 再继续 `watchlist` 与 `news`
