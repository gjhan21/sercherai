# 阶段9：股票智能量化选股系统与后台独立模块

最后更新: 2026-03-21
状态: Planned（文档已收口，可直接进入第一轮）

## 阶段目标

让 `strategy-engine` 从“接收候选种子后再排序”的股票 MVP，升级为“由 Python 自主生成股票池、自动挖掘候选种子、融合量化因子 / 事件图谱 / 多角色推演后输出最终组合”的智能选股系统；同时把这套能力在 Admin 管理后台中落成独立一级模块，可独立配置、独立运行、独立评估、独立审核、独立发布。

这一阶段要解决的核心问题有两个：
- 当前股票链路仍偏向“Go 先给候选，Python 再评分”，还不是完整的自动化选股系统
- 当前管理侧能力分散在市场中心、策略作业中心、配置页里，还没有独立的研究 / 配置 / 运行 / 评估工作台

## 对应需求

- Python 不能再默认依赖人工 `seed_symbols` 或固定候选，而是要自己从全市场里找出候选股票
- 选股不能只看轻量量化分，要吸收事件、关系结构和多角色观点
- Admin 后台要新增独立模块，单独管理智能选股的配置、运行、评估、审核与发布
- 参数要可配置、可版本化、可回滚、可审计
- 未来的“为什么选这只股票”要能回答到因子、事件、角色、世界推演和风险边界层

## 核心结果

- 股票主链从“种子排序器”升级为“自动股票池构建 + 自动种子挖掘 + 融合决策器”
- Admin 新增独立一级菜单 `智能选股`，不再把功能散落在 `MarketCenter` / 通用作业中心
- 后台具备从配置、运行、候选查看、评估、审核到发布的完整闭环
- 事件图谱、多角色推演和投资逻辑记忆开始成为正式能力，不再只是后置 explanation 装饰层
- 选股结果具备版本、证据、参数、推演和后续表现的统一复盘能力

## 本阶段边界

本阶段做：
- 股票自动股票池与自动种子挖掘方案落地
- 金融本体、事件图谱、关系增强与投资逻辑记忆规划
- 多角色推演从 explanation 层升级为决策增强层
- Admin 独立模块的信息架构、配置体系、运行中心、评估中心、审核发布中心规划
- 配置版本化、审计、实验和运行产物归档规划

本阶段不做：
- 社交媒体式大规模社会仿真
- 自建重型图数据库平台
- 自动实盘交易
- 高频实时策略
- 一开始就让多角色完全主导打分、取代量化主骨架

## 阶段9收口决策

为了避免后续开发中反复回到方案层，这一阶段统一按以下口径执行：

- Admin 新增独立一级模块 `智能选股`，不再继续把新能力挂回 `MarketCenterView` 或通用作业中心页。
- 第一轮只上线一个默认 profile：`profile_default_stock_auto`。趋势型、稳健型、事件型等多 profile 模板进入阶段9后续轮次，不与第一轮并做。
- `selection_mode` 默认值固定为 `AUTO`。`seed_symbols` 从默认主链退出，只保留给 `MANUAL / DEBUG` 模式使用。
- 图谱首版不引入独立图数据库，也不单开图服务；统一复用现有标准化市场真相源，在运行期派生轻量关系证据和快照表。
- 评估首版只做线上前瞻评估摘要与受控历史重跑，不做完整历史回测平台，不做复杂策略研究工作台。
- 审核与发布首版继续复用现有 `strategy-engine` 发布模型、发布审计和发布归档链路；`智能选股` 模块只补专用入口、摘要与候选视图，不新造第二套发布系统。
- Python 继续负责选股计算主链；Go 后端继续作为真相源、调度、持久化与 Admin API 唯一出口；Admin 不直接调用 Python 服务。
- 多角色推演在阶段9首版中仍是“增强层 + veto 层”，量化主骨架仍然是最终排序的主体。

## 为什么要做这一阶段

前 8 个阶段已经让我们具备了：
- 股票 / 期货独立策略服务
- explanation / 版本历史 / 发布复盘
- 多源行情与资讯真相源
- 管理端作业中心与配置能力

但股票方向仍然有三个明显短板：
- Python 还没有真正掌控“从全市场自动选股”的主链
- 量化评分、事件理解、角色推演之间还没有形成完整融合决策
- 管理端缺少一个真正服务研究、配置、运行、评估、审核的独立模块

所以阶段9不是简单再补一个页面，而是把“股票策略如何被构建、管理、验证和发布”整体升级。

## 从 MiroFish 借鉴什么

本阶段会借鉴 `MiroFish` 的方法论，但不会照搬其社交媒体仿真技术栈。

重点借鉴的优点：
- 先把输入材料转成结构化世界模型，而不是直接让大模型给结论
- 先定义本体、实体和关系，再做后续推演和报告
- 把角色画像做成一等公民，让不同角色拥有不同视角和偏好
- 把并行世界推演做成正式分析层，而不是最后补一句“bull / bear”
- 把推演过程中的新结论回写为记忆，形成可持续演化的判断链
- 报告生成不只拼接文案，而是“先规划 -> 工具检索 -> 分段生成 -> 留工具轨迹”

对我们来说，对应关系应该是：
- “种子材料” -> 市场真相源、公告、新闻、政策、产业链、商品 / 期货联动信号
- “本体 / 图谱” -> 股票、公司、行业、概念、政策、事件、商品、期货、产业链节点
- “Agent 人设” -> 基本面研究员、趋势交易员、资金流交易员、事件研究员、风险官、反方审查员
- “并行世界” -> Base / Bull / Bear / Policy / Earnings / Commodity Shock

## 新的股票主链

阶段9默认把股票主链正式升级为以下 9 层：

### 1. Universe Builder

Python 每日从 Go 后端内部真相源读取全市场可选股票快照，先完成基础过滤：
- 停牌 / ST / 退市整理
- 上市天数不足
- 成交额 / 成交量过低
- 价格异常
- 数据缺失
- 行业或股票黑白名单

### 2. Auto Seed Miner

系统自动从全市场里挖掘候选种子，不再默认依赖人工传入股票代码。

首版固定支持 5 类自动种子：
- 趋势种子：突破、动量、回撤修复、均线结构
- 资金种子：净流入、量比放大、成交额异常
- 估值 / 质量种子：估值改善、盈利稳定、资产质量
- 事件种子：公告、政策、研报、新闻热度
- 联动种子：行业轮动、商品价格、期货结构、产业链联动

### 3. Factor Engine

量化主骨架继续保留，但从“候选打分器”升级为“正式因子引擎”。

首版固定分成 6 组因子：
- 趋势与动量
- 波动与回撤
- 流动性与成交
- 资金流
- 估值与质量
- 资讯与事件

### 4. Regime Engine

先识别当前市场状态，再决定不同 profile 或权重模板：
- TREND
- DEFENSIVE
- RISK_OFF
- EVENT_DRIVEN

### 5. Financial Ontology & Market Graph

构建金融本体和轻量市场图谱，至少覆盖以下实体：
- `Stock`
- `Company`
- `Industry`
- `ConceptTheme`
- `Index`
- `Commodity`
- `FuturesContract`
- `Policy`
- `Event`
- `ResearchReport`
- `SupplyChainNode`

至少覆盖以下关系：
- `BELONGS_TO`
- `BENEFITS_FROM`
- `AFFECTED_BY`
- `SUPPLIES_TO`
- `COMPETES_WITH`
- `CORRELATED_WITH`
- `CONFIRMS_SIGNAL`
- `WEAKENS_SIGNAL`
- `TRIGGERS_ROTATION`
- `CONNECTED_TO_FUTURES`

### 6. Agent Panel 2.0

多角色不再只做 explanation，而是正式进入决策链。

首版固定角色：
- 基本面研究员
- 行业景气研究员
- 技术趋势交易员
- 资金流交易员
- 事件驱动交易员
- 宏观 / 政策观察员
- 风险控制官
- 反方审查员

### 7. Multi-World Simulation

对每日 Top 候选进行多世界推演，首版固定世界模板：
- `BASE`
- `BULL`
- `BEAR`
- `POLICY_POSITIVE`
- `EARNINGS_DISAPPOINT`
- `COMMODITY_SHOCK`

### 8. Decision Fusion Engine

最终分数由四层融合构成：
- `QuantBase`
- `EventGraph`
- `AgentConsensus`
- `RegimeFit`

再叠加统一 `RiskPenalty`。

默认口径：
- 量化主骨架至少保留 60% 以上权重
- 多角色推演只做增强和 veto，不直接替代量化主链

### 9. Memory & Thesis Evolution

每只股票都要沉淀投资逻辑记忆：
- 首次入选原因
- 被强化的证据
- 被削弱的证据
- 角色观点如何变化
- 逻辑何时失效
- 后续表现如何

## Admin 独立模块规划

### 模块定位

Admin 侧新增独立一级菜单：
- `智能选股`

它不再挂在 `MarketCenter` 或通用策略作业中心下面，而是成为单独模块。

职责边界固定如下：
- `MarketCenter`: 继续负责数据源同步、行情 / 资讯准备、市场运营检查台、数据新鲜度
- `Strategy Engine`: 继续负责通用作业、发布、归档、版本历史、解释快照
- `智能选股`: 负责股票池、因子、图谱、角色、运行、评估、审核、发布

### 模块信息架构

一级模块 `智能选股` 的完整信息架构固定如下：

1. `总览`
2. `运行中心`
3. `策略配置`
4. `股票池规则`
5. `因子与权重`
6. `事件图谱`
7. `多角色推演`
8. `候选与组合`
9. `评估与回测`
10. `审核与发布`

### 每个页面要承接什么

### 1. 总览

一进模块先能看到：
- 今日运行状态
- 最新交易日与数据新鲜度
- 当前默认策略 profile
- 当前识别出的市场状态
- 今日候选池数量与最终入选数量
- 最近一次运行耗时
- 最近一次发布结果
- 最近 5 / 10 / 20 日命中率与超额摘要
- 当前 warning / veto / 数据异常提醒

### 2. 运行中心

专用 job center，服务智能选股全流程：
- 手动发起运行
- 定时运行
- 历史交易日重跑
- 按 profile 重跑
- 查看阶段耗时、日志和失败原因
- 查看产物快照
- 比较两次运行结果

### 3. 策略配置

统一管理 profile：
- 新建 / 复制 / 停用 / 启用
- 默认 profile 切换
- 草稿 / 发布版本
- 配置 diff
- 回滚
- profile 标签

### 4. 股票池规则

把 universe 管理从因子页分离出来：
- 当前可选股票总数
- 被过滤原因分布
- 行业 / 板块覆盖
- 黑白名单
- 某只股票为何被排除
- universe 配置模拟与对比

### 5. 因子与权重

把量化主骨架做成正式配置中心：
- 因子组 / 子因子列表
- 权重编辑
- 窗口参数
- 因子贡献度
- 因子命中分布
- 不同市场状态下的权重模板
- 单只股票因子拆解

### 6. 事件图谱

负责查看和维护事件增强层：
- 今日核心事件
- 股票 / 行业 / 商品 / 期货的关联路径
- 事件证据源
- 图谱边强弱
- 手工修正或失效标记
- 重点事件录入

### 7. 多角色推演

负责角色、世界模板与推演结果：
- Agent 列表与权重
- 角色配置
- 共识规则
- veto 规则
- 多世界模板
- 某只股票的角色分歧
- 某角色的历史表现

### 8. 候选与组合

负责查看“系统到底选了谁”：
- 当日候选池 Top N
- 最终组合 Top N
- 每只股票的因子 / 图谱 / 推演 / 风险拆解
- 入选原因与未入选原因
- 板块 / 风格分散度
- 人工剔除与人工补充 override

### 9. 评估与回测

负责回答“系统到底好不好用”：
- 5 / 10 / 20 日收益
- 超额收益
- 命中率
- 最大回撤
- 风险收益比
- 分市场状态表现
- 分 profile 表现
- 分因子变更表现
- 分角色 veto 表现
- A / B profile 对照

### 10. 审核与发布

模块内独立承接审核与发布：
- 待审核推荐
- 自动生成报告
- 风险提醒 / warning / veto 摘要
- 通过 / 驳回 / override
- 发布备注
- 历史发布记录
- 已发布版本对比

## 兼容与迁移策略

### 任务协议兼容

- 股票任务 payload 走“只增不改”策略，新增 `selection_mode`、`universe_scope`、`profile_id`、`debug_seed_symbols`、`min_listing_days`、`min_avg_turnover`，不移除现有 `seed_symbols`、`market_scope`、`excluded_symbols`、`limit`、`min_score`。
- 新旧字段并存期，Python 侧以 `universe_scope` 为优先真相源，若缺失则回退 `market_scope`；Go 后端在过渡期同时写出两者，直到所有消费方切换完成。
- `seed_symbols` 在协议层继续保留，但默认 `AUTO` 模式下必须允许其为空；只有 `MANUAL / DEBUG` 模式才读取该字段。

### 数据与存储兼容

- `stock_recommendations`、`stock_reco_details`、现有 publish record、客户端 explanation 结构继续保留，不因阶段9第一轮而改写客户端主消费契约。
- 新增数据对象统一走增量建表，首批默认包含：`stock_selection_profiles`、`stock_selection_profile_versions`、`stock_selection_runs`、`stock_selection_run_stage_logs`、`stock_selection_run_candidates`、`stock_selection_run_portfolio`、`stock_selection_publish_reviews`。
- 图谱与角色相关数据首版优先落在运行快照和证据表中，不要求一开始就抽成独立基础设施。

### 后台迁移兼容

- `智能选股` 上线后，旧 `MarketCenter` 中与股票量化相关的入口先保留为历史入口或跳转入口，直到新模块覆盖旧消费路径后再移除。
- 第一轮不调整客户端菜单和用户侧页面结构；只要最终发布仍回写现有推荐表，客户端不需要同步改版。
- 新权限码全部按增量方式加入，不移除现有 `market.view` 等旧权限，避免阻断已有后台账号访问。

### 默认初始化策略

- 数据库迁移中默认物化一套 `profile_default_stock_auto` 配置，并把它标记为默认 profile。
- 若环境中尚无智能选股权限绑定，后台默认把 `stock_selection.view` 与 `stock_selection.manage` 先加入管理员角色，细粒度权限在后续轮次补齐。

## 风险点

- 如果让多角色过早主导排序，会削弱可回测性和可解释性
- 如果阶段9一开始就做重型图数据库，会显著拖慢主链交付
- 如果后台模块继续散落在旧页面里，后续运营和研究会非常难用
- 如果不做配置版本化和审计，后续调参会无法复盘
- 如果股票自动种子挖掘仍依赖少量高成交额候选，系统会继续保留“半自动选股”局限
