# 阶段6：后台策略管理协同

最后更新: 2026-03-18
状态: Done

## 阶段目标

让 `strategy-engine` 从“可调用的独立服务”升级为“可被后台管理和运营配置的策略中台”。

这阶段重点解决一个问题：
当前服务已经能生成结果，但 Admin 与 Go 后端对种子输入、多角色模拟、场景模板和发布策略几乎没有管理能力，导致“服务在跑，后台却不能管”。

## 对应需求

- 后端管理要同步跟上，不能只有服务在跑
- 管理端要能参与管理：多角色模拟、角色定义、场景模板、风控阈值、发布策略

## 核心结果

- Admin 可管理策略版本、种子集、角色定义、场景模板、发布策略
- Go 后端补齐管理接口、持久化模型和审计链路
- `strategy-engine` 的运行结果可在后台被查看、回放、比对、人工干预和再次发布

## 本阶段边界

本阶段做：
- 后台配置与管理能力
- Go 后端模型、接口、持久化、审计能力
- Admin 管理页面和操作流程
- `strategy-engine` 配置读取协议与运行时消费

本阶段不做：
- 实时实盘交易
- 大规模 agent 社会模拟
- 复杂研究平台化权限系统重构

## 需要补齐的后台能力

### 1. 策略种子管理

目标：让运营/研究可以决定“本次策略输入了什么”。

建议管理对象：
- 股票种子池
- 期货合约池
- 新闻/事件种子条目
- 板块/主题标签
- 排除名单
- 每日默认输入模板

建议后台能力：
- 查看当前生效 seed set
- 新建/复制/停用 seed set
- 预览某个 seed set 影响的标的范围
- 对每日任务选择指定 seed set 运行

建议落库对象：
- `strategy_seed_sets`
- `strategy_seed_set_items`

## 2. 多角色配置管理

目标：让后台能管理“有哪些角色参与模拟，以及各自职责是什么”。

建议管理对象：
- 角色编码，例如 `TREND`、`EVENT`、`LIQUIDITY`、`RISK`、`BASIS`
- 角色名称与说明
- 角色启停状态
- 角色权重
- 角色输出字段约束
- 角色 veto 权限

建议后台能力：
- 查看角色面板
- 配置角色参与范围（股票 / 期货 / 全部）
- 调整角色权重和 veto 开关
- 标记默认角色组合

建议落库对象：
- `strategy_agent_profiles`
- `strategy_agent_profile_versions`

## 3. 场景模板管理

目标：让后台能管理 bull / base / bear / shock 的触发规则和解释模板。

建议管理对象：
- 场景编码
- 场景名称
- 场景触发阈值
- 分数调整规则
- 风险提示模板
- 推荐动作模板

建议后台能力：
- 调整场景阈值
- 维护股票 / 期货不同模板
- 预览某一标的在各场景下的解释输出

建议落库对象：
- `strategy_scenario_templates`
- `strategy_scenario_rules`

## 4. 发布与风控策略管理

目标：让后台能管理“什么结果可以发，什么结果必须拦”。

建议管理对象：
- 最大风险等级
- 最小置信度
- veto 后处理策略
- 发布前检查清单
- 默认 reviewer / publisher
- 版本命名与发布说明模板

建议后台能力：
- 配置发布门槛
- 查看某次任务为什么被拦截
- 人工 override 并记录原因
- 人工补充运营说明后再发布

建议落库对象：
- `strategy_publish_policies`
- `strategy_publish_reviews`

## 5. 作业运行与复盘管理

目标：让后台看到每次 job 是怎么跑出来的，而不是只有最终结果。

建议后台能力：
- job 列表
- job 详情
- 输入快照
- 角色输出面板
- 场景输出面板
- 风险否决记录
- 发布记录与版本对比
- replay 与人工备注

建议落库对象：
- `strategy_job_runs`
- `strategy_job_artifacts`
- `strategy_job_agent_outputs`
- `strategy_job_replays`

## Admin 页面建议

建议在 `/Users/gjhan21/cursor/sercherai/admin/src/views/MarketCenterView.vue` 基础上拆分独立管理区块：
- 策略总览
- 种子管理
- 角色配置
- 场景模板
- 发布策略
- 作业中心
- 版本复盘

建议体验原则：
- 不要把所有东西继续堆在一个 tab 内
- 优先把“运行”和“配置”拆开
- 每个配置页都要能看到“影响范围”和“最近生效结果”

## 验收标准

- 后台可查看并管理种子集
- 后台可查看并管理角色定义、权重和 veto 能力
- 后台可查看并管理场景模板和发布策略
- 后台可查看 job 运行详情、角色输出、场景输出和 replay
- 后台可人工发布、拦截或补充说明，并有审计记录
- Go 后端与 Admin 的字段、接口、权限说明同步更新

## 风险点

- 如果配置项过多却没有默认模板，后台会变成复杂控制面板，运营无法使用
- 如果不做版本化，角色和模板修改后会影响历史结果解释
- 如果不记录运行时快照，后台看到的配置会和历史 job 当时的真实配置脱节

## 建议拆分顺序

1. 作业中心与详情页
2. 种子管理
3. 角色配置
4. 场景模板
5. 发布策略与人工 override

## 当前执行切片

本轮先落最小可感知闭环：
- 先做种子管理
- 再做多角色配置管理
- 再把 Admin 接到现有策略中心，支持查看、新建、编辑默认配置
- 生成每日股票 / 期货策略时，优先消费后台默认启用的 seed set / agent profile
- 然后补作业中心，让后台看到任务输入、运行状态和结果摘要

- 本轮仍未引入新表持久化运行时快照，但已经把“后台能读懂一次任务怎么跑出来”这个最小闭环补齐

第五轮已经落地以下内容：

- 股票 / 期货每日生成链路在拿到最终 job detail 后，会立即把 `payload_snapshot`、`config_refs`、`publish_policy_preview`、`report_snapshot`、`warnings` 与统计信息归档到 Go 后端
