# 阶段8：真实数据接入与真相源增强

最后更新: 2026-03-20
状态: In Progress

## 阶段目标

让 `strategy-engine` 从“基于内置样本和既有快照可解释运行”升级为“基于真实行情、资讯、事件和统一真相源稳定运行”。

这一阶段重点解决两个问题：
- 当前股票 / 期货策略仍大量依赖内置样本、已有推荐表和回填快照，真实市场数据接入还没正式成为主链路
- Admin 虽然已经具备作业中心、发布复盘和顶层摘要，但后续如果继续扩列表或运营面板，仍需要更稳定的数据源与摘要模型支撑

## 对应需求

- 由 Go 后端统一接入第三方行情、资讯、事件、资金流等数据
- `strategy-engine` 只消费本地标准表，不直接绑定外部供应商
- 继续增强本地真相源，让客户端 explanation、Admin 复盘、后台摘要都建立在同一套内部数据之上

## 核心结果

- 股票 / 期货策略开始消费真实行情与资讯数据，而不是主要依赖内置样本
- Go 后端形成统一的市场数据标准表、同步任务、健康检查和数据新鲜度指标
- `strategy-engine` 与外部供应商解耦，只依赖内部标准化数据接口
- Admin 可查看数据源状态、同步结果和关键摘要，不需要继续回退到 runtime 临时数据

## 本阶段边界

本阶段做：
- 行情 / 资讯 / 事件 / 资金流等真实数据接入规划与落地
- Go 后端统一标准表、同步任务和数据源健康检查
- `strategy-engine` 对内部真相源的数据消费改造
- Admin 侧数据源与同步摘要展示
- 后续还需要的顶层稳定摘要补充

本阶段不做：
- 高频实时撮合或实盘交易
- 策略研究平台全面重构
- 复杂的多供应商自动切换平台
- 一开始就接入过多数据商并做重型抽象

## 为什么要先做这一阶段

前 7 个阶段已经完成了：
- 策略服务骨架
- 股票 / 期货 MVP
- 多角色与场景模拟
- 发布与复盘
- 后台配置协同
- 客户端 explanation 与版本历史

但如果没有真实数据接入，策略体系仍有两个明显短板：
- “解释链路”已经很完整，但输入源还不够真实、稳定、可持续
- “后台和前台的真相源”已经开始收口，但市场数据本身还没有完全纳入统一真相源

所以阶段8不是继续做页面，而是回到最根部，把“输入真相源”补齐。

## 需要补齐的能力

### 1. 真实行情接入

目标：让股票 / 期货策略有稳定的价格、成交量、涨跌幅、基差等真实输入。

建议优先接入：
- 股票日线行情
- 指数基准行情
- 期货主力合约行情
- 基础成交额 / 换手 / 持仓类指标

建议原则：
- 统一由 Go 后端拉取
- 统一落本地标准表
- `strategy-engine` 不直接请求第三方接口

### 2. 资讯与事件接入

目标：让 explanation 里的事件面、新闻面、主题面有真实来源。

建议优先接入：
- 快讯 / 新闻
- 公告 / 政策摘要
- 板块 / 主题事件
- 与标的映射后的事件标签

建议原则：
- 先做“够用的结构化摘要”，不一开始追求全量 NLP 平台
- 优先服务策略种子、场景解释和风险提示

### 3. 内部标准表与同步任务

目标：让所有策略输入先进入统一真相源，再供上层消费。

建议补齐：
- 市场数据标准表
- 资讯事件标准表
- 数据源同步任务
- 数据新鲜度检查
- 同步失败告警 / 日志 / 回补机制

### 4. strategy-engine 消费层改造

目标：让服务只面向内部标准表，而不是面向某个外部供应商。

建议改造：
- seed ingest 改成从内部真相源读取
- feature factory 改成基于标准表计算
- report / explanation 明确标注数据时间和来源
- 为后续多供应商切换留接口，但第一版不做过度抽象

### 5. Admin 数据源与摘要增强

目标：让后台不仅能看策略结果，也能看“数据有没有准备好”。

建议补齐：
- 数据源列表与健康状态
- 最近一次同步时间
- 同步成功 / 失败统计
- 数据新鲜度提醒
- 如果策略依赖数据缺失，要能在作业中心或发布链路里明确提示

## 建议接口与对象方向

建议优先围绕已有后端能力继续补：
- 数据源健康检查接口
- 行情同步任务接口
- 资讯同步任务接口
- 最近同步摘要接口
- 作业中心的数据依赖状态字段

建议优先复用或增强现有对象：
- `data_sources`
- `data_source_health_logs`
- `stock_market_quotes`
- 新闻 / 事件相关表
- 作业中心顶层 summary 字段

## 建议拆分顺序

1. 先固化数据契约与标准表，不急着全量接入
2. 先跑通股票 / 指数 / 期货的最小行情同步闭环
3. 再跑通新闻 / 事件的最小结构化闭环
4. 再让 `strategy-engine` 从内部真相源消费真实数据
5. 最后补 Admin 数据源摘要与剩余顶层稳定字段

## 第一轮执行切片建议

本轮先做最小可感知闭环：
- 先明确股票 / 期货行情的标准输入字段
- 先确定第一优先级数据源和同步方式
- 先把“Go 后端落标准表 -> strategy-engine 消费内部数据”这条主链画清楚
- 暂时不同时推进太多供应商

## 验收标准

- 至少一条真实行情同步链路可跑通
- `strategy-engine` 至少一个核心输入开始消费内部标准表而非内置样本
- Admin 能看到数据源状态、同步时间和异常摘要
- explanation / report 能标注关键数据时间与来源
- 文档、表结构、同步任务、消费链路保持一致

## 风险点

- 如果一开始就同时接多个供应商，复杂度会明显上升
- 如果不先定义统一标准表，后面会再次把策略层绑死在外部 API 上
- 如果数据新鲜度和失败重试不做最小闭环，策略结果会变得不可解释
