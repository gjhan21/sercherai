## 推荐实施顺序

推荐严格按以下顺序推进：

1. 包0：基线校验与目录准备
2. 包1：协议与数据库骨架
3. 包2：Python 自动选股主链
4. 包3：Go 后端持久化与 Admin API
5. 包4：Admin 独立模块骨架
6. 包5：审核发布闭环与迁移期兜底

原因：
- 没有协议和数据库骨架，后续链路会持续返工
- 没有 Python 自动主链，后端和后台只能做空壳
- 没有 Go 持久化与接口，Admin 页面会被迫直接读运行时嵌套结果
- 没有迁移期兜底，第一轮很容易把旧后台入口打断

## 联调顺序

推荐联调按以下顺序执行：

1. 执行数据库迁移，确认新表和默认 profile 已就位
2. 启动 `strategy-engine`，本地验证 `AUTO` 模式任务可跑
3. 启动 Go 后端，验证新 Admin 接口与旧生成接口可共存
4. 启动 Admin，验证新菜单、四页骨架、基础交互
5. 通过 Admin 发起 run，观察 run summary、候选、审核、发布
6. 发布完成后，验证旧消费链路仍能读取结果

## 数据库迁移执行要点

迁移必须遵守以下原则：
- 只增不改，不删除旧表，不收缩旧字段
- 新表、索引、默认值都走可重复执行的 `IF NOT EXISTS` 风格
- 默认 profile 与默认权限如需落库，采用幂等写法
- 如需要补系统配置或权限种子，迁移脚本必须可重复执行

建议检查项：
- 新表主键、唯一键、按 `run_id / profile_id / trade_date / status` 的查询索引
- 候选和组合表是否按 `run_id + symbol + stage` 提供唯一约束
- review 表是否允许审核备注、审核人和审核时间为空后再补齐
