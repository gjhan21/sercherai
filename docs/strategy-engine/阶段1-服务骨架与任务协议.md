# 阶段1：服务骨架与任务协议

最后更新: 2026-03-17
状态: Done

## 阶段目标

建立 `strategy-engine` 的最小可运行服务骨架，让系统先具备“可启动、可创建任务、可查询状态”的基础能力。

## 核心结果

- `services/strategy-engine/` 目录可启动
- 提供 health 接口
- 提供 job 创建 / 查询接口
- 定义任务状态、基础日志、配置文件结构

## 主要改动点

- 创建服务目录与基础依赖
- 建立 `app/main.py`、`settings.py`
- 建立 `api/routes_health.py`、`api/routes_jobs.py`
- 建立 `schemas/job.py`
- 建立基本任务状态：QUEUED / RUNNING / SUCCEEDED / FAILED

## 验收标准

- 服务可本地启动
- `GET /internal/v1/health` 返回正常
- `POST /internal/v1/jobs/...` 能返回 `job_id`
- `GET /internal/v1/jobs/{job_id}` 能返回状态
- 文档与接口结构一致

## 风险点

- 如果阶段1直接混入股票/期货逻辑，会让基础骨架不稳定
- 如果 job 协议设计不清晰，后续阶段会重复返工

## 实际完成范围

本阶段实际采用了“FastAPI + 内存任务仓 + BackgroundTasks 模拟执行”的轻量方案：
- 先固定 API 契约和任务状态流转
- 先让服务具备可启动、可验证、可集成的生命体
- 真实队列、数据库持久化、复杂调度器延后到后续阶段

## 验收结果

- 本地安装依赖并通过测试：`pytest`，3 个用例全部通过
- 本地启动冒烟验证：`uvicorn app.main:app --host 127.0.0.1 --port 8081`
- 健康检查接口返回正常：`GET /internal/v1/health`

## 偏差说明

- 阶段1未引入 Celery / Arq，而是先用内存任务仓稳定协议
- 阶段1未引入 MySQL / Redis 持久化，以避免过早绑定基础设施
- 阶段1未接入现有 Go 后端，只先完成独立服务可运行闭环

## 遗留问题

- 任务状态目前仅保存在进程内，重启后会丢失
- 任务执行仍是模拟执行，尚未进入真实策略管线
- 还未定义 Go 后端调用该服务的 client / adapter
- 还未建立统一错误码、鉴权与内部签名机制
