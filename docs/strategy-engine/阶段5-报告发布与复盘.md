## 实际完成范围

本阶段实际采用了“内部发布归档 + 轻量复盘”的最小实现：
- job 成功后可通过 `POST /internal/v1/publish/jobs/{job_id}` 生成发布记录
- 发布记录同时输出 Markdown 和 HTML 报告，便于 Admin 或 Go 后端消费
- 发布归档会保留版本号、关键标的、发布 payload、报告快照和 replay 信息
- 已提供 `GET /internal/v1/publish/history/{job_type}` 查看历史版本
- 已提供 `POST /internal/v1/publish/compare` 对比两个发布版本的标的变化、payload 变化与风控差异
- 已提供 `GET /internal/v1/publish/records/{publish_id}/replay` 回放 warnings、veto 与失效条件摘要
- Go 后端与 Admin 已接入股票 / 期货 publish record 详情查询，可直接查看 HTML / Markdown 报告正文、publish payload 与 report snapshot

## 偏差说明

- 阶段5优先做“能发布、能复盘、能对比”，不提前做复杂 BI 系统
- 当前发布归档为内存实现，先保证链路跑通，不提前引入对象存储或数据库归档
- `go_backend_publisher` 当前产出的是可供 Go 后端消费的发布包和结构化 payload，暂未在 `strategy-engine` 内直接发起远程数据库写入
- Admin 侧页面现已接入股票 / 期货发布报告详情预览，本轮仍未补导出、分享和持久化归档能力

## 遗留问题

- 发布记录仍是内存态，服务重启后不会保留，需要后续接入数据库或对象存储
- 版本对比目前聚焦标的变化与警告差异，尚未细化到逐字段 diff
- Go 后端消费 publish record 的正式 adapter 仍可继续补强，尤其是期货侧回写闭环
- Admin UI 已支持股票 / 期货两侧的 Markdown / HTML 报告详情预览；后续可继续补导出与分享动作
