# 阶段7：客户端策略解释展示

最后更新: 2026-03-18
状态: Done

## 阶段目标

让客户端从“只展示推荐结果”升级为“展示推荐结论背后的证据、过程与边界条件”，让用户知道：
- 为什么选择这只股票或这个期货策略
- 系统参考了哪些种子输入
- 多角色模拟分别看到了什么
- 我们做了哪些大量工作，最终才得出这个结论

## 对应需求

- 前端（客户端）需要展示种子输入、多角色模拟等信息
- 要告诉客户为什么选择这个股票
- 要把系统做过的工作、分析视角和结论形成多角度提示

## 核心结果

- 推荐详情页可展示“输入 -> 分析 -> 模拟 -> 风险 -> 结论”的完整链路
- 用户能看到种子输入、图谱摘要、场景模拟、多角色观点和失效条件
- 页面表达从“结果页”升级为“信任页”

## 本阶段边界

本阶段做：
- Client 页面信息架构升级
- 后端补齐解释型字段与详情接口
- 推荐卡片、详情页、历史页的解释化表达
- 客户理解路径与信任增强设计

本阶段不做：
- 对话式智能投顾
- 自动个性化推荐引擎重构
- 复杂回测图表平台
- 真实行情与资讯数据接入

## 用户要看到的五层信息

### 1. 结论层

先告诉客户结论，不让客户迷路：
- 推荐动作
- 适用周期
- 仓位建议
- 风险等级
- 核心一句话理由

### 2. 证据层

展示“为什么会选中它”：
- 关键种子输入
- 技术面 / 资金面 / 事件面 / 情绪面摘要
- 行业/主题联动点
- 当前市场环境标签

### 3. 模拟层

展示“如果从不同视角看，会得到什么判断”：
- bull / base / bear / shock 四场景卡片
- 每个场景下的 thesis、动作建议、主要风险
- 多角色模拟面板：趋势、事件、流动性、风险、基差
- 共识动作与分歧点

### 4. 风险层

告诉客户“什么情况下这结论会失效”：
- 失效条件
- 不建议参与的情形
- 止盈 / 止损 / 观察位
- veto 或警告原因

### 5. 工作量层

把系统背后的工作显性化，让客户感知价值：
- 当日处理了多少种子
- 经过了多少轮筛选
- 哪些角色参与了评审
- 生成了多少场景和对比
- 当前策略版本与更新时间

## 文案与产品表达原则

- 先讲结论，再讲过程
- 不要堆技术词，要把“种子输入”“角色模拟”翻译成用户能理解的话
- 同时展示机会与风险，避免像单向营销页
- 一定要写明失效条件，信任来自边界感
- 多角度提示要短、密、可扫读，不要一次塞成长篇报告

## 验收标准

- 用户在客户端能直接看到“为什么推荐”
- 用户能看到至少 3 类不同视角的证据提示
- 用户能看到多角色模拟摘要与场景推演摘要
- 用户能看到风险、失效条件和操作边界
- 首页、详情页、历史页形成一致表达，不互相打架
- 页面构建通过，移动端和桌面端都能正常展示

## 风险点

- 如果把解释内容写得过长，用户只会跳过不看
- 如果只展示正向理由，不展示边界和失效条件，会削弱信任
- 如果后台没有先把种子、角色、场景结构化，前端会很难稳定消费

## 建议拆分顺序

1. 先补后端 explanation 字段
2. 首页卡片增强
3. 推荐详情页策略档案化
4. 历史页 / 归档页解释增强
5. 会员页或关注页承接“继续追踪”动作

## 与阶段6的依赖关系

客户端策略解释展示依赖阶段6先把后台和后端配置能力补齐。

原因很简单：
- 后台不先管理好 seeds / agents / scenarios，前端展示出来的内容就会不稳定
- 没有运行快照和版本信息，客户端很难讲清“为什么今天是这个结论”
- 只有先把后台控制面和后端真相源建好，客户端展示才能长期可信
