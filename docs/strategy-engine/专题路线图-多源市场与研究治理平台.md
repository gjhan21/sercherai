# 专题路线图：多源市场与研究治理平台

最后更新: 2026-03-23  
状态: Planned

> 基线说明：本路线图建立在阶段8、阶段9已收官的基础上，当前代码基线为 commit `27ac324`。它不再回退为新的“阶段10”，而是把后续 6 个独立专题拆成一个总路线图，按专题文档和专题收官方式推进。

## 为什么单独立专题

阶段8、阶段9已经把双资产研究平台、真实数据主链、图谱代理、审核发布、统一 explanation 消费收口完成。

当前剩余工作已经不再适合继续塞回阶段文档，原因有三点：

- 范围已经跨越数据治理、研究建模、后台工作台、回测评估、通知审计和旧链路收缩，属于多个独立子系统
- 各专题的交付节奏、依赖关系和验收口径不同，继续混写会导致文档漂移
- 后续实现应以“一个专题一个真相源文档”的方式推进，而不是重新发明新的大阶段

因此，后续默认采用：

- 一份总路线图
- 六个独立专题
- 三轮主线交付
- 每个专题单独收官、单独回写

## 总目标

把当前“可运行的双资产研究平台”升级为“可治理、可审计、可扩多供应商、可深化事件语义、可深化期货主数据、可完整评估、可逐步收缩旧链路”的研究与运营底座。

最终目标不是再做一轮大而全重构，而是形成以下 6 类稳定能力：

1. 多源市场数据统一模型与供应商治理平台
2. 股票事件图谱与资讯语义研究工作台
3. 期货深度因子与主数据画像平台
4. 回测、实验与自动化运营联动平台
5. 默认策略自动落库审计通知与系统消息平台
6. 旧入口、旧页面、旧链路收缩平台

## 约束与默认约定

- 继续保持“只增不改”兼容策略，旧客户端主消费契约、旧入口、旧调度兜底路径在专题收官前不删除
- 人工审核发布继续作为默认发布方式，不启用自动发布
- `strategy-engine` 继续只消费 Go 后端统一真相源，不直接绑定第三方供应商
- 图服务继续维持“可降级、不阻塞 run 与发布链路”的原则
- 每个专题进入开发前，必须先新增专题文档，不回到阶段8/9正文里混入长期蓝图

## 专题拆解

### 专题A：多源市场数据统一模型与供应商治理

目标：

- 把当前“多个供应商已经接入、但仍偏运行时拼接”的状态，升级成统一市场数据模型、统一路由、统一质量、统一回退和统一供应商治理平台

收官标准：

- 股票、期货、资讯、主数据都具备统一 provider registry、capability 声明、优先级路由、fallback 策略和质量画像
- truth 构建不再依赖每条链路各自拼接 source priority，而是统一走供应商治理层
- Admin 数据源管理从“操作台”升级为“治理台”：支持能力矩阵、默认源、备源、失败路由、质量分和治理建议
- 下游上下文、同步作业、truth 重建、研究运行都能回读统一的供应商治理摘要

不并入第一轮的内容：

- 实时 tick 级撮合
- 商业数据采购流程
- 自动计费与供应商成本中心

建议交付物：

- 统一 `market data domain model`
- provider capability schema
- provider routing / fallback policy
- source quality scoring / freshness scoring
- Admin 数据治理页
- 供应商审计日志与异常画像

### 专题B：股票事件图谱、资讯语义与事件审核台

目标：

- 把当前 run 级图快照与基础实体关系，升级为“股票事件图谱 + 资讯语义抽取 + 事件审核台”的专题研究底座

收官标准：

- 资讯同步后可沉淀标准化事件、主题、行业、公司、政策、财报、供应链等实体与关系
- 图谱不只承接 run 快照，还能承接“事件真相源”和“事件审核状态”
- Admin 新增事件审核台，支持事件聚类、标签修正、误判回收和主题关系校正
- 股票候选详情可读取“事件证据链”，而不只是当前 run 的浅层 related entities

不并入第一轮的内容：

- 全自动语义标注零人工审核
- 舆情情绪量化的高频生产化
- 面向 C 端的开放图谱探索社区

建议交付物：

- 资讯语义抽取与事件标准化 schema
- 事件审核台
- 行业 / 主题 / 公司 / 政策关系增强
- 股票候选事件证据链增强
- 图谱误差回收与再训练样本池

### 专题C：期货深度因子与主数据画像

目标：

- 把当前“可运行的期货策略研究链”继续升级为“具备库存、交割地、品牌、等级、跨品种 spread、多市场联动”的深层期货研究底座

收官标准：

- 期货主数据具备商品、交割地、品牌、等级、仓库、库存口径、期限结构、跨期/跨品种联动的统一画像
- 研究因子可稳定使用库存变化、仓单变化、期限结构偏移、品牌/等级溢价、跨市场联动指标
- 期货候选与组合页能读到更细的因子证据，而不是只显示趋势/风险摘要
- 图谱中能表达商品链、替代关系、上下游成本传导和政策冲击路径

不并入第一轮的内容：

- 复杂期权波动率曲面
- 全市场 CTA 实盘执行平台
- 高频日内研究链

建议交付物：

- futures instrument master v2
- 交割地 / 品牌 / 等级 / 仓单 / 库存画像
- spread / basis / term structure 深化因子
- 跨商品 / 跨市场联动指标
- 期货深层 evidence cards

### 专题D：回测平台、实验平台与自动化运营联动

目标：

- 把当前已有的评估摘要、A/B 实验埋点、运营节奏工具升级为完整研究评估与增长联动平台

收官标准：

- 研究运行可关联回测配置、基准、样本窗口、交易成本假设、命中率、回撤、行业暴露等完整回测报告
- 实验平台具备 cohort、归因、订单级唯一去重、模板表现对比和多页面 journey 分析
- 运营工具不再只是任务与看板，而是与策略发布、用户分群、实验结果和通知系统打通
- Admin 能以“研究效果 -> 用户承接 -> 会员转化 -> 留存反馈”的链路查看平台效果

不并入第一轮的内容：

- 自动化广告投放平台
- 通用 BI 替代品
- 任意页面任意可编排工作流引擎

建议交付物：

- research backtest service / report
- evaluation warehouse
- growth experiment v2
- cohort / attribution / order dedupe
- 运营节奏与策略效果联动看板

### 专题E：默认策略自动落库审计通知与系统消息平台

目标：

- 把当前散落在审核、发布、任务中心和 workflow 的消息能力，升级为统一的审计通知与系统消息平台

收官标准：

- 默认策略落库、强制发布、驳回、回滚、truth 重建失败、供应商切换、回测异常等都能进入统一审计事件流
- 后台支持规则化通知订阅、消息聚合、去重、升级和确认闭环
- 审核中心、任务中心、流程消息不再各自维护一套弱耦合通知语义
- 支持“研究事件、数据事件、发布事件、运营事件”四类消息统一模型

不并入第一轮的内容：

- 第三方 IM 全渠道编排平台
- 对外开放的 webhook 市场
- 全量 SOC / SIEM 替代系统

建议交付物：

- audit event schema
- notification routing policy
- system inbox / digest / escalation
- Admin 消息中心重构
- 默认策略自动落库审计链

### 专题F：旧入口、旧页面、旧链路收缩

目标：

- 在兼容策略稳定的前提下，逐步收缩 `MarketCenter`、旧每日生成入口、旧研究配置页和旧客户端 fallback 的冗余路径

收官标准：

- `MarketCenter` 只保留过渡说明或彻底迁出，不再继续承接核心研究动作
- 旧每日生成入口转为明确的兜底或调度适配层，不再与新研究平台并行演进
- 旧客户端 fallback 只保留必要兼容，不再承接新字段缺口
- 文档、路由、权限、菜单、任务和日志口径统一，不再出现新旧双入口都能修改同一核心对象

不并入第一轮的内容：

- 一次性大切换
- 强制删除所有旧接口
- 不可回退迁移

建议交付物：

- legacy matrix 与收缩顺序
- route / permission / menu 收缩清单
- fallback 收缩清单
- 迁移开关与回退策略

## 推荐主线顺序

这 6 个专题不建议并行全开。推荐顺序如下：

### 第一轮：底座治理

- 专题A：多源市场数据统一模型与供应商治理
- 专题E：默认策略自动落库审计通知与系统消息平台

为什么先做：

- A 决定后续股票事件、期货深层因子和回测数据能否建立在稳定真相源之上
- E 决定后续专题的审计、消息和运营动作有没有统一事件流

### 第二轮：研究深化

- 专题B：股票事件图谱、资讯语义与事件审核台
- 专题C：期货深度因子与主数据画像

为什么第二轮做：

- 这两项都依赖更稳定的数据治理底座
- 它们分别代表股票和期货研究价值的下一跳，不应互相阻塞，但应共享治理底座和图谱基础设施

### 第三轮：平台收口

- 专题D：回测平台、实验平台与自动化运营联动
- 专题F：旧入口、旧页面、旧链路收缩

为什么最后做：

- D 需要前面研究深化后的稳定证据和产物，才值得搭完整评估平台
- F 只能在新能力足够稳定、消息审计完善后再做，不适合提前动刀

## 推荐先开工专题

推荐先从专题A开始。

原因：

- 这是股票事件图谱、期货深层因子、回测平台和旧链路收缩的共同依赖
- 当前多源已经可用，但仍处于“能跑、能切源、能回退”的工程阶段，尚未真正形成治理平台
- 先把供应商治理收口，后面研究层的语义与因子深化才不会继续散落在各条链路里

专题A收官后，再进入专题E，把审计和通知统一起来；这两者结合后，后续专题的开发成本会显著下降。
