# 推荐股票网站开发工作流

最后更新: 2026-03-16
状态: 生效中

## 目标

确保后续开发严格遵循 Vibe Coding 节奏，避免边做边散、做完无记录。

## 开发前动作

每次开始开发前，必须完成以下检查：

1. 阅读 `docs/vibe-stock-growth/README.md`
2. 阅读当前阶段文档
3. 确认本次只做当前阶段范围内事项
4. 确认验收标准与回写要求
5. 如有范围变更，先改文档，再写代码

## 开发中动作

- 优先复用现有能力，不轻易重写已有链路
- 每一轮只做一个最小可感知闭环
- 同时关注三件事：用户价值、信任表达、可验收性
- 若实现与阶段文档不一致，必须先更新文档再继续

## 开发后动作

每次完成一个可提交节点后，必须回写：
- 完成了什么
- 哪些地方与计划不一致
- 还遗留哪些问题
- 下一阶段是否可开始

建议在对应阶段文档中补充以下内容：
- 实际改动文件
- 实际完成范围
- 验收结果
- 风险与后续建议

## 阶段状态定义

- `Planned`: 已规划，未开始
- `In Progress`: 正在开发
- `Blocked`: 有依赖或风险，暂时阻塞
- `Done`: 功能已完成并完成文档回写
- `Deferred`: 暂缓执行，等待后续阶段

## 范围控制原则

以下事项不应在早期阶段提前扩张：
- 复杂社群系统
- 高复杂度实时推送体系
- 完整量化回测引擎重构
- 大规模后台重构

优先级更高的是：
- 首页价值感知
- 推荐可信度
- 用户回访理由
- 后台生命周期可管理

## 文档更新模板

可在每个阶段文档末尾保持以下结构：
- 当前状态
- 已完成
- 偏差说明
- 遗留问题
- 进入下一阶段条件
